# 夜间代码分析任务

## 执行时间
每日 02:00 自动执行

## 任务清单

1. **代码分析**
   - 扫描 `src/` 识别未被调用的函数
   - 检测功能重复的 API（参数/逻辑相似度 > 70%）
   - 识别超长函数（> 100 行）和高复杂度函数（圈复杂度 > 15）

2. **文档审计**
   - 对比 `docs/` 与实际实现的一致性
   - 标记过时/错误的描述
   - 列出缺失文档的新功能

3. **生成报告**
   - 输出路径: `docs/core/tasks/2026-03-02-nightly-analysis/reports/2026-03-02-report.md`
   - 历史报告按日期归档，便于追溯

## 执行原则

- **自动执行**: 无需人工干预
- **只读模式**: **严禁修改任何代码或文档**，仅生成审计报告
- **报告驱动**: 所有发现的问题必须记录在报告中，由人工根据报告决定是否修改
- **自动归档**: 按日期保存历史报告到 `docs/core/tasks/2026-03-02-nightly-analysis/reports/`
- **失败处理**: 记录错误到 `docs/core/tasks/2026-03-02-nightly-analysis/logs/` 但不中断流程

## 重要约束 🚫

**禁止修改任何文件**

- 不要修改代码文件（.js, .ts, .py 等）
- 不要修改文档文件（.md, .txt 等）
- 不要修改配置文件（.json, .yaml, .env 等）
- **唯一允许的操作：创建新的报告文件和日志文件**

**报告必须清晰可执行**

审计报告必须让其他开发者能够：
1. 清楚知道存在什么问题
2. 准确定位问题位置（文件路径、行号）
3. 理解问题的严重程度和影响
4. 知道如何修复（提供具体建议）

## 分析范围

### 代码目录
- `server/` - 后端服务代码
- `lib/` - 公共库代码
- `models/` - 数据模型代码
- `tools/` - 工具脚本代码
- `frontend/` - 前端代码（需要检查前端是否调用后端 API）

### 设计文档
- `docs/design/` - 所有设计文档
- 重点关注 v1/ 和 v2/ 目录
- 对比文档描述与实际代码实现
